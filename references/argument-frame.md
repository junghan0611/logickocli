# 분석 출력 포맷

이 스킬이 한국어 자연어 논증을 받았을 때 출력해야 하는 구조화된 frame.

## 기본 frame (default)

```
[주장]
사용자가 결론으로 내세우는 명제.

[전제]
P1. ...
P2. ...
(가능하면 사용자 발화 안의 전제 표시어 — "왜냐하면", "이므로", "→" — 를 보고 추출)

[결론]
C. ...

[형식화]
P1: A → B
P2: A
C:  B
(symbol/variable assignment 명시: 예 A = "비가 온다", B = "길이 젖는다")

[사용 규칙]
PROP.MODUS_PONENS / 전건긍정 / modus ponens / MP

[숨은 가정]
- ...
(전제로 명시되지 않았지만 결론이 따라 나오기 위해 필요한 명제)

[평가]
- 형식적 타당성: 타당 / 부당
- 건전성: 평가 불가 (전제 사실 검토 필요) / 건전 / 불건전
- 비형식 오류 후보: 없음 / FALLACY.XXX

[더 분석할 수 있는 곳]
(생략 가능; 예: "P1을 양상 형식으로 보면 □(A → B)도 가능"; "Lean으로 형식화 가능"; "베이지언으로 다시 보면 ...")
```

## 변형 frame

### proof_mode

연역적 증명을 작성·검증하라는 요청일 때.

```
[목표]
증명하려는 결론.

[가정]
- 명시적 가정
- 사용 가능한 정리

[증명 단계]
1. ...        (used rule: PROP.XXX)
2. ...        (used rule: PRED.XXX)
3. ⊢ goal     (used rule: ...)

[정리보조기 후보]
- Lean tactic: ...
- Coq tactic: ...
(번역만; 실제 검증은 별도)
```

### debate_mode

쟁점이 있는 주장 — "당신은 A라고 했고 나는 B라고 한다" 류.

```
[주장 1]  사용자 측

[주장 2]  반대 측

[공유 전제]
- ...

[쟁점이 되는 전제]
- ...

[가장 강한 논증]
- ...

[가장 약한 단계]
- ...

[steelman]
(상대 입장을 가능한 한 강하게 재구성)

[잠재적 반례]
- ...
```

### modal_mode

양상 주장 ("필연", "가능", "반드시", "어쩌면", "증명할 수 있다" 등)이 핵심.

```
[양상 주장]
원문.

[양상 식별]
□ / ◇ / □◇ / ◇□ / ...

[체계 후보]
K / T / S4 / S5 / GL 중 어디서 평가하는가?

[표준 양상 표기]
...

[해당 체계에서의 평가]
정리인가? 반례 있는가? 알려진 동치는?
```

### probability_mode

확률·믿음·베이즈가 핵심.

```
[주장]
"P(H|E)는 ...다" 형태로 정규화.

[관련 양]
P(H), P(E), P(E|H), P(E|¬H)

[베이즈 갱신]
P(H|E) = P(E|H)·P(H) / P(E)

[추정의 약점]
- 기저율 무시 가능성
- 표본 크기
- 사전확률 가정
```

## 응답 언어 규칙

- 기본: 한국어 (canonical 한자말) + 기호.
- 영어 병기: 사용자 영어 입력 시, 또는 표준어가 영어로 더 명확할 때.
- 한말 (이고 없애기, 차근차근 이끌기 등): 한국어 토착화 작업의 결과물이지만, 이 도구는 LLM 호환을 위해 한자말로 응답한다. 사용자가 한말로 들어오면 매핑은 인식하되 응답에서는 canonical로 정규화한다.
- 외부 저자가 정의한 자체 기호 체계는 해당 체계의 정의·공리를 검산하는 별도 도구가 필요하다. 이 도구는 표준 양상 기호 (□/◇) 와 그 의미론에 한정한다.

## 출력 길이 가이드

- 짧은 발화에는 짧은 frame. 모든 섹션을 다 채우려 하지 말 것. 빈 섹션은 생략.
- 복잡한 다단계 증명 외에는 [형식화]·[사용 규칙]·[평가] 세 줄이면 충분한 경우도 많다.
- 모드 진입 신호가 약할 때는 한 줄 요약 후 "구조화 분석 원하시면" 식의 옵트인.
